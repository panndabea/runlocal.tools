Local tools for private files.

runlocal.tools is an open toolbox for everyday file and media tasks — designed to run entirely in your browser.

Open a tool, drop a file, process it locally, and download the result. No upload step. No server-side processing. No account required.

The long-term goal is a Unix-like toolbox for the web: small, focused tools that can be shared by URL and composed into local browser-native pipelines.

Why local

Everyday file tasks should not require blind trust.

Many simple web utilities still require users to upload private files to remote servers. For image, data, and media processing, that is often unnecessary: modern browsers are capable enough to do the work locally.

🔐

No upload step

Core workflows process files directly on the user's device. The file does not need to be transferred to a third-party service.

🖥️

Browser-native execution

Tools use browser APIs, Web Workers, Streams, Canvas, and eventually WASM to perform useful work in the client runtime.

🧭

Clear, stable routes

Each tool has a simple URL such as /images/merge, /images/resize, or /csv/to-json.

Core idea

Share tools, not files.

Instead of sending data to remote processing services, users can share links to local tools. The recipient opens the tool and runs it in their own browser, on their own device, with their own files.

That small shift changes the default relationship between web utilities and user data. The useful functionality becomes shareable while the private data remains local.

Privacy first

Users stay in control of their data.

runlocal.tools avoids unnecessary accounts, uploads, opaque backends, and hidden retention. If a task can run locally, it should run locally.

  • Private files stay on the user's device.
  • Processing logic is inspectable and open source.
  • Tools are reusable, self-hostable, and extension-friendly.
  • The platform is built as public digital infrastructure.

How it works

Simple, direct, local.

The basic interaction should feel obvious: open a URL, choose a file, process it in the browser, and keep control of the output.

1

Open source

Inspectable, reusable, and self-hostable.

runlocal.tools is intended as an open project, not a closed utility service. The code, processing logic, and user-facing tools should be transparent and reusable by others.

Open by default

The platform is meant to be inspectable and understandable, with clear routes, clear behavior, and no hidden remote processing behind the scenes.

Reusable primitives

Shared browser-side processing components can support multiple tools, reducing duplication and making the system easier to extend.

Self-hostable

Organizations, communities, and individuals should be able to run their own version of the toolbox when they need control over deployment.

Digital commons

Practical utilities should be accessible without ads, opaque infrastructure, unnecessary lock-in, or unnecessary upload steps.

FAQ

Clear principles, plain answers.

Does runlocal.tools upload files?

No. The core model is local execution in the browser. Files are processed on the user's device rather than handed to a remote processing service.

What does “share tools, not files” mean?

A user shares a URL to a tool, not the data itself. The recipient opens the same tool and processes their own local files in their own browser.

How are pipelines different from normal web apps?

Each tool is a small focused unit. Future workflows can connect these units so an output from one local tool becomes the input for another, similar to Unix pipes.

Are receipts the main product?

No. The main project is an open local-first toolbox. Receipts are an optional layer for automated workflows, auditability, or compliance-sensitive environments.
